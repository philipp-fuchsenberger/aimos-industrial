# Open Agent Memory Format (OAMF) — v1.0 Draft

**A universal, framework-agnostic format for AI agent memory portability.**

*Proposed by AIMOS Project (Fuchsenberger, 2026)*
*License: Apache 2.0*

---

## Abstract

AI agents build long-term memory through interactions — facts, preferences, events,
learned patterns. Currently, this memory is locked into proprietary formats per framework.
When switching platforms, all accumulated knowledge is lost.

OAMF defines an open, human-readable format that any AI agent framework can import/export.
An agent's memory becomes portable — like moving contacts between phones.

---

## 1. Design Principles

1. **Human-readable**: JSONL (one JSON object per line) — inspectable with any text editor
2. **Framework-agnostic**: No dependency on any specific agent framework, DB, or LLM
3. **Tiered**: Supports importance scoring, recency, access frequency — not just flat key-value
4. **Categorized**: Semantic (facts), Episodic (events), Procedural (rules/preferences)
5. **Provenanced**: Every memory records its source (user, agent, tool, system)
6. **Incrementally mergeable**: Can be appended, deduplicated, and merged across exports

---

## 2. File Format

### 2.1 Memory File: `*.oamf.jsonl`

Each line is a self-contained JSON object representing one memory entry:

```jsonl
{"v":1,"key":"user_name","value":"Ugur Şeker","cat":"semantic","imp":9,"acc":15,"src":"user","created":"2026-03-21T10:04:00Z","updated":"2026-03-21T10:52:00Z","meta":{}}
{"v":1,"key":"firma","value":"Aich Makalsan - Beton Makine İmalatı","cat":"semantic","imp":8,"acc":7,"src":"agent","created":"2026-03-21T10:49:00Z","updated":"2026-03-21T10:49:00Z","meta":{"derived_from":"web_search"}}
{"v":1,"key":"meeting_q2","value":"Q2 Budget 50K genehmigt, Deadline 30.04.","cat":"episodic","imp":7,"acc":3,"src":"user","created":"2026-03-20T14:00:00Z","updated":"2026-03-20T14:00:00Z","meta":{"participants":["Fuchsenberger","Yılmaz"]}}
{"v":1,"key":"response_style","value":"User prefers bullet points, max 2 sentences","cat":"procedural","imp":6,"acc":20,"src":"agent","created":"2026-03-18T09:00:00Z","updated":"2026-03-21T12:00:00Z","meta":{}}
```

### 2.2 Field Definitions

| Field | Type | Required | Description |
|---|---|---|---|
| `v` | integer | yes | Format version (currently `1`) |
| `key` | string | yes | Unique identifier for this memory (snake_case recommended) |
| `value` | string | yes | The memory content (plain text, any language) |
| `cat` | string | yes | Category: `semantic`, `episodic`, `procedural` |
| `imp` | integer | yes | Importance: 1 (trivial) to 10 (critical) |
| `acc` | integer | yes | Access count: how often this memory was retrieved |
| `src` | string | yes | Source: `user`, `agent`, `tool`, `system`, `auto`, `import` |
| `created` | string | yes | ISO 8601 timestamp (UTC) of first creation |
| `updated` | string | yes | ISO 8601 timestamp (UTC) of last modification |
| `meta` | object | yes | Extensible metadata (framework-specific, can be `{}`) |

### 2.3 Categories

| Category | Description | Examples |
|---|---|---|
| `semantic` | Facts, knowledge, static information | Names, preferences, business data, contacts |
| `episodic` | Events, interactions, time-bound | Meetings, conversations, decisions, milestones |
| `procedural` | Rules, patterns, learned behaviors | Communication style, workflow preferences |

### 2.4 Source Types

| Source | Description |
|---|---|
| `user` | Explicitly shared by the user ("remember that...") |
| `agent` | Derived by the agent from context |
| `tool` | Result of a tool call (web search, file read, etc.) |
| `system` | Auto-generated by the framework (consolidation, migration) |
| `auto` | Auto-extracted from conversation without explicit instruction |
| `import` | Imported from another system |

## 3. Agent Package: `*.oap` (Open Agent Package)

A complete portable agent is a ZIP archive with extension `.oap`:

```
agent_name.oap (ZIP):
├── manifest.json           # Package metadata
├── agent.yaml              # Agent definition (OSSA-compatible)
├── memory.oamf.jsonl       # Long-term memory (this format)
├── history.jsonl           # Conversation history (optional)
├── system_prompt.txt       # Core personality prompt
└── skills.json             # Skill configuration and state (optional)
```

### 3.1 manifest.json

```json
{
  "format": "oap",
  "version": "1.0",
  "agent_name": "kral",
  "display_name": "Kral",
  "created": "2026-03-21T10:00:00Z",
  "exported_from": "AIMOS v4.3.0",
  "memory_entries": 47,
  "history_entries": 156,
  "language": "tr",
  "checksum": "sha256:..."
}
```

### 3.2 agent.yaml (OSSA-compatible)

```yaml
apiVersion: oap/v1
kind: Agent
metadata:
  name: kral
  displayName: Kral
  language: tr
  version: "1.0"
spec:
  personality:
    description: "Şeker ailesinin dijital asistanı. Galatasaray fanatik taraftarı."
    style: delikanlı
    tone: respectful-but-casual
  capabilities:
    skills:
      - brave_search
      - file_ops
      - scheduler
      - football_observer
      - tr_calendar_awareness
    connectors:
      - telegram
      - email
    interAgent:
      enabled: true
  security:
    editableCredentials:
      - EMAIL_ADDRESS
      - EMAIL_PASSWORD
```

### 3.3 history.jsonl (optional)

```jsonl
{"role":"user","content":"Selam Kral","timestamp":"2026-03-21T10:04:09Z","channel":"telegram","sender_id":"978404520"}
{"role":"assistant","content":"Selam Abi! KRAL burada! 👋","timestamp":"2026-03-21T10:04:34Z"}
```

## 4. Scoring & Relevance

Importing systems SHOULD use the tiered scoring fields for memory injection:

```
score = imp * recency_weight * (1 + ln(acc + 1))

where:
  recency_weight = 1.0 / (1.0 + days_since_updated * 0.1)
```

This ensures that important, recent, frequently-accessed memories are prioritized
when the context window is limited.

Systems that don't support tiered scoring can fall back to sorting by `imp` descending.

---

## 5. Merge & Deduplication

When importing memories into an existing agent:
- **Same key**: newer `updated` timestamp wins (upsert)
- **Different key, similar value**: Jaccard similarity > 0.6 on tokenized values → merge
  (take higher `imp`, sum `acc`, newer timestamp)
- **Source preservation**: imported memories get `src: "import"` with original source in `meta`

---

## 6. Privacy & Security

- Memory files MAY contain PII — handle with the same care as the agent's database
- Export tools SHOULD offer a `--anonymize` flag that runs PII detection before export
- The `meta` field can contain a `pii_classes` array: `["name", "email", "phone"]`
- Importing systems SHOULD respect PII markings and apply their own vault/anonymization

## 8. Reference Implementation

AIMOS provides the reference implementation:

```bash
# Export agent with full memory
python -m aimos.export --agent kral --output kral.oap

# Export memory only
python -m aimos.export --agent kral --memory-only --output kral.oamf.jsonl

# Import agent to new system
python -m aimos.import --file kral.oap

# Merge memory into existing agent
python -m aimos.import --file kral.oamf.jsonl --agent kral --merge

# Export with PII anonymization
python -m aimos.export --agent kral --anonymize --output kral_clean.oap
```
